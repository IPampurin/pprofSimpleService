## Файлы для pprofSimpleService - пример оптимизации простого API-сервиса с профилировкой  

### 📋 Описание проекта  

**pprofSimpleService** - демонстрационный HTTP-сервис полного цикла оптимизации с использованием  
стандартных инструментов профилирования: `pprof`, `go test -bench`, `benchstat` и `trace`.  

Сервис предоставляет три простых API-эндпоинта:  
- сложение двух чисел (`/sum`),  
- вычисление числа Фибоначчи (`/fib`),  
- выделение и немедленное освобождение блока памяти (`/allocate`).  

Изначально функция `Fib` реализована рекурсивно (экспоненциальная сложность), что создаёт высокую  
нагрузку на CPU. Функция `Allocate` нагружает сборщик мусора. Это кандидаты для поиска узких мест  
через профилирование и последующей оптимизации.  

В рамках проекта проведены:  
- замеры производительности (бенчмарки),  
- захват CPU, memory и trace профилей,  
- замена рекурсивного алгоритма Фибоначчи на итеративный,  
- количественное сравнение результатов через `benchstat`.  

Проект может служить шаблоном для внедрения практик профилирования в любой Go-микросервис.  

### 🖥️ Возможности  

- **HTTP API** с тремя эндпоинтами (см. ниже).  
- **Встроенный pprof-сервер** для сбора профилей в рантайме (`/debug/pprof/`).  
- **Бенчмарки** для изолированного тестирования CPU и памяти.  
- **Готовые команды** для снятия профилей и их визуализации.  
- **Демонстрация оптимизации** с численными результатами.  
- **Docker Compose** для быстрого развёртывания.  
- **Простой веб-интерфейс** (HTML/CSS/JS) для ручного тестирования.  

### 📡 HTTP-методы (API)  

- `GET /sum?a=<int>&b=<int>` - возвращает сумму двух целых чисел.  
- `GET /fib?n=<int>` - возвращает n-е число Фибоначчи (n от 1 до 45).  
- `POST /allocate` - тело `{"size": <int>}` - выделяет `size` байт и сразу освобождает, возвращает количество выделенных байт.  
- `GET /debug/pprof/*` - стандартные эндпоинты pprof.

### 🚀 Быстрый старт  

Требования:  

    Docker и Docker Compose (либо Go 1.21+ для локального запуска)  
    Свободный порт: 8081 (определяется HTTP_PORT)  

Запуск через Docker:  

```bash
docker compose up --build
```

Локальный запуск (без Docker):  

```bash
go run ./cmd
```

После запуска сервис доступен по адресу:  

    http://localhost:8081  

Остановка:  

```bash
docker compose down
# или Ctrl+C для локального запуска
```

### ⚙️ Конфигурация  

Все настройки задаются через файл .env в корне проекта.  

    HTTP_HOST=0.0.0.0            # хост (в докере - имя контейнера)
    HTTP_PORT=8081               # порт сервера
    HTTP_READ_TIMEOUT=15s        # таймаут чтения запроса
    HTTP_WRITE_TIMEOUT=15s       # таймаут записи ответа
    HTTP_IDLE_TIMEOUT=60s        # keep-alive
    HTTP_SHUTDOWN_TIMEOUT=30s    # таймаут graceful shutdown

### 🔬 Профилирование и оптимизация  

1. Запуск бенчмарков (baseline)  

Бенчмарки изолированно тестируют бизнес-функции, минуя HTTP-слой.  

```bash
# запуск всех бенчмарков с замером аллокаций, 5 прогонов
go test -bench=. -benchmem -count=5 ./internal/service/ > old.txt
```

2. Снятие профилей до оптимизации  

CPU-профиль (для Fib):  

```bash
go test -bench=BenchmarkFibN40 -cpuprofile=cpu.out ./internal/service/
go tool pprof -http=:8080 cpu.out
```

Профиль памяти (для Allocate):  

```bash
go test -bench=BenchmarkAllocate10MB -memprofile=mem.out ./internal/service/
go tool pprof -http=:8080 mem.out
```

Трассировка выполнения:  

```bash
go test -bench=BenchmarkFibN40 -trace=trace.out ./internal/service/
go tool trace trace.out
```

3. Оптимизация  

Исходная (медленная) реализация Fib:  

    func (s *Service) Fib(n int) int64 {
        if n <= 0 { return 0 }
        if n == 1 { return 1 }
        return s.Fib(n-1) + s.Fib(n-2)
    }

Оптимизированная (итеративная) реализация:  

    func (s *Service) Fib(n int) int64 {
        if n <= 0 { return 0 }
        if n == 1 { return 1 }
        var a, b int64 = 0, 1
        for i := 2; i <= n; i++ {
            a, b = b, a+b
        }
        return b
    }

4. Повторные замеры и сравнение  

После замены кода повторяем бенчмарки:  

```bash
go test -bench=. -benchmem -count=5 ./internal/service/ > new.txt
```

Устанавливаем benchstat (если нет):   

```bash
go install golang.org/x/perf/cmd/benchstat@latest
```

Сравниваем результаты:  

```bash
benchstat old.txt new.txt
```

**Выводы:**  

- Оптимизация Fib дала колоссальный прирост производительности:  
Для n=40 время выполнения сократилось с 614 мс до 14 нс (ускорение ~43 миллиона раз).  
Аллокации памяти отсутствуют в обеих реализациях (рекурсия не выделяла память на куче, только стек).  
Бенчмарки для n=20 и n=30 также показали ускорение на 100% (фактически время стало ничтожно малым).  

- Функция Allocate не подвергалась изменениям, поэтому значительного изменения производительности ожидать не следовало.  
Отклонения (+33% для 10 МБ) вызваны скорее всего локальными возмущающими факторами (вызваны фоновой активностью ОС (сборка мусора, прерывания)).  
При повторных запусках значение может колебаться.  
Количество аллокаций и объём выделяемой памяти не изменились (по 1 аллокации на вызов).  

- Приведённые отклонения в пределах менее 5% носят характер статистической погрешности и могут в расчёт не приниматься.
